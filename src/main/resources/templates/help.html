Overview

PLM Migration Assistant documents source-to-Windchill mappings, transformation requirements, lookup/crosswalk needs, validation notes, customer decisions, assumptions, exclusions, and exports. It does not infer mappings, recommend mappings, use AI, or generate executable transformation scripts.

Project dashboard

The dashboard summarizes a migration project, target object tabs, source imports, open audit items, open decisions, and generated documentation. Use the dashboard to open the target-first mapping workspace, run deterministic audit checks, and create exports. The dashboard only shows unresolved items from the latest audit run because a new audit run clears previous audit items before creating the current set.

Real project workflow

For a real migration project, start with files exported from your source PLM system and a JSON definition of the Windchill target model/staging schema. No backend setting is required for the basic MVP flow when you use the dashboard upload forms. H2 is used by default for local testing; use PostgreSQL through Docker Compose when you want project data to persist in a local database.

  1. Create a project from the project list.
  2. Export source tables from Agile or another PLM system as CSV files. For Agile, typical exports are ITEM, BOM, DOCUMENTS, and ATTACHMENTS.
  3. On the project dashboard, upload each CSV in the Import Agile/source CSV card. The MVP imports and profiles one CSV at a time.
  4. Prepare a Windchill target model JSON file with target object types and attributes. You can use src/main/resources/demo-data/windchill_target_model.json as a shape/example.
  5. Upload that JSON in the Import Windchill target model JSON card. The mapping workspace tabs are created from these records.
  6. Open the target-first mapping workspace, select a Windchill target tab, and manually enter mappings and requirements.
  7. Run deterministic audit checks, resolve required unmapped fields and customer decisions, then generate and download exports. If errors are resolved and only warnings remain, the project status changes from Blocked to Needs Review.

API alternatives are also available: use POST /api/projects/{projectId}/source-imports/csv with multipart form field file for CSV imports and POST /api/projects/{projectId}/target-model/import-json with multipart form field file for target model JSON.

Import source CSV

Use the dashboard CSV upload to import source system exports. The MVP expects a header row and comma-separated values. It creates a source import record, a source table, and source column profile rows. The profile shows row count, column count, null percentage, distinct count, max observed length, sample values, duplicate indicators, and likely list/enumeration indicators.

This profiling is factual only. It does not infer mappings and it does not recommend target fields. The architect still chooses every source-to-target mapping manually.

Import Windchill target model JSON

Use target model JSON to configure Windchill target object tabs and fields. The JSON should contain a model name/version and a targetObjectTypes array. Each object type can include object type name, display name, soft type, staging name, sort order, and an attributes array.

Each target attribute can include:

  • attributeName and displayName
  • dataType
  • required
  • maxLength
  • legalValues
  • defaultValue and exampleValue
  • description, validation notes, and staging notes
  • sortOrder

After import, open the mapping workspace. The target object type tabs come from the imported records, not hardcoded UI tabs.

Source schema browser

The source schema browser lists imported source tables/files and their discovered columns. It shows deterministic profile facts, including sample values, null percentage, distinct count, maximum observed length, duplicate indicators, and likely list/enumeration indicators. These facts help an architect make a manual mapping decision; they are not automatic mapping suggestions.

Mapping workspace

The workspace is organized by configurable Windchill target object types. Each tab is loaded from a TargetObjectType record. Select a target tab, review the required Windchill fields, and manually select a source field, mapping type, status, owner, default value, transformation requirement, and validation notes.

Mapping matrix and status

The center matrix is the system of record for target fields in the selected Windchill object type. After you save a mapping, the row remains populated with its selected source field, mapping type, mapping status, owner, default value, and transformation notes.

  • Blocked/red: required target field has no source, default value, or not-applicable status.
  • GREEN: mapping status is approved.
  • YELLOW: mapping needs customer review.
  • PURPLE: mapping type is lookup.
  • BLUE: mapping type is default value.
  • GRAY: mapping is not applicable or not migrated.

Detail and audit panel

The right panel explains status colors and summarizes existing mappings for the selected target object tab. It is a quick audit and review area; the editable fields in the center matrix are the primary place to update mapping data in the MVP.

Exports

Exports are generated from structured data already entered in the application. They document mapping specifications, customer review workbooks, validation plans, JSON mapping definitions, and implementation requirements. They do not generate production SQL, executable transformations, or AI-generated pseudocode.

In the current MVP, export buttons create a DocumentExport record in the database. The dashboard then shows a Download button for each generated export. Files ending in .md are Markdown documents: plain-text documentation that can be opened in a text editor, browser, GitHub, or converted to Word/PDF in a later phase. The JSON mapping export downloads as a .json file.

Demo how-to: Agile ITEM to Windchill WTPart

  1. Open the project list and click Create Agile-to-Windchill Demo Project.
  2. Open the generated project dashboard.
  3. Click Open Target-First Mapping Workspace.
  4. Select the WTPart target object tab.
  5. In the source schema browser, review the ITEM fields such as ITEM_NUMBER, DESCRIPTION, and LIFECYCLE_PHASE.
  6. In the number row, select ITEM.ITEM_NUMBER, choose DIRECT, choose APPROVED, add validation notes, and save.
  7. In the name row, select ITEM.DESCRIPTION, choose DIRECT, choose PROPOSED, add a transformation comment if needed, and save.
  8. In the lifecycleState row, select ITEM.LIFECYCLE_PHASE, choose LOOKUP, choose NEEDS_CUSTOMER_REVIEW, and note that the lifecycle crosswalk requires customer approval.
  9. Confirm that saved source fields, mapping types, statuses, owners, defaults, and notes remain visible in the center matrix.
  10. Run the deterministic audit from the project dashboard.
  11. Export the Mapping Specification, Customer Review Workbook, and Implementation Requirements Export.

Troubleshooting mapping status display

If a row stays red after saving, check whether the required row has at least one of these deterministic resolution values: a selected source field, a default value, or a NOT_APPLICABLE mapping status. Also check that the saved mapping appears in the row's Current Mapping column. The input controls should now remain populated after save because the workspace reloads saved mapping data by target attribute.
